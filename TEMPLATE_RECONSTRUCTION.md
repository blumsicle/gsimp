# Generated Template Reconstruction Guide

This document describes the starter project generated by `gsimp create` closely enough that a fresh Codex session could rebuild a very similar project from scratch in an empty directory.

It documents the generated scaffold, not the `gsimp` generator implementation.

## Purpose

The generated project is a Go CLI starter with these goals:

- keep `main` thin
- centralize parser and logger wiring in a reusable internal package
- keep app config explicit and typed
- support YAML config plus CLI flag overrides
- support linker-injected build metadata
- provide a Makefile with standard local workflows
- include smoke tests for the CLI wiring most likely to break

## Stack

- CLI parser: `github.com/alecthomas/kong`
- logging: `github.com/rs/zerolog`
- YAML decoding: `github.com/goccy/go-yaml`
- tests: `github.com/stretchr/testify`

## Generated Layout

The generated package layout is:

- `cmd/<project>` for the binary entrypoint and command tree
- `cmd/globals.go` for shared injected CLI globals
- `internal/appconfig` for typed app config defaults, loading, and tests
- `internal/cli` for parser/build-info/logger runtime helpers

Why this split exists:

- `internal/cli` is framework-style runtime code, not app config
- `cmd/globals.go` stays importable by subcommands, unlike package `main`
- `internal/appconfig` keeps app config out of the `cmd` namespace while still staying available across the module

## Config Model

The generated scaffold has two config-related types.

### `appconfig.Config`

Defined in `internal/appconfig/config.go` and `internal/appconfig/load.go`.

Current fields:

- `RootPath string        `yaml:"root_path"``
- `GitLocation string     `yaml:"git_location"``
- `LogLevel zerolog.Level `yaml:"log_level"``

Current defaults come from:

- `func Default() *Config`
  now named `func Default() *Config`

Those defaults are:

- `RootPath = "."`
- `GitLocation = ""`
- `LogLevel = zerolog.InfoLevel`

YAML loading is handled by:

- `func (c *Config) LoadYAML(path string) error`

Current behavior of `LoadYAML`:

- missing config file is not an error
- YAML is decoded into the existing config object, so defaults remain unless overridden
- `RootPath` and `GitLocation` are passed through `os.ExpandEnv(...)` after decode

That means values like `root_path: $HOME/src` work as expected.

### `cmd.Globals`

Defined in `cmd/globals.go`.

Current fields:

- `ConfigFile string`

Current Kong tags:

- short flag: `-c`
- long flag: `--config-file`
- default: `~/.config/<project>/config.yaml`
- type: `path`
- help: `Path to the config file`

## CLI Runtime Layer

The generated shared runtime code lives in:

- `internal/cli/buildinfo.go`
- `internal/cli/runner.go`

### `internal/cli/buildinfo.go`

Defines:

- `type BuildInfo struct { Name, Version, Commit string }`

### `internal/cli/runner.go`

Defines:

- `type Config struct { Description string; BuildInfo BuildInfo }`
- `func Options(cfg Config) []kong.Option`
- `func New(app any, cfg Config, options ...kong.Option) (*kong.Kong, error)`
- `func Parse(app any, cfg Config, options ...kong.Option) *kong.Context`
- `func NewLogger(level zerolog.Level) zerolog.Logger`
- `func Run(ctx *kong.Context, log zerolog.Logger, args ...any) error`

Important runtime behavior:

- `Options` sets Kong name, description, compact help, and the interpolated version string
- `Parse` parses `os.Args[1:]`
- `Parse` calls `parser.FatalIfErrorf(err)` on parse failure
- `NewLogger` returns a console logger writing to `os.Stderr`
- the returned logger has `.Level(level)`, timestamp, and `logger=main`
- `Run` prepends the logger to `ctx.Run(...)`

## Root Binary Structure

The generated binary lives in `cmd/<project>`.

### `cmd/<project>/main.go`

Responsibilities:

- define linker-overridable build vars:
  - `name`
  - `version`
  - `commit`
- create `appConfig := appconfig.Default()`
- create the root CLI struct
- build the shared CLI runtime config
- call `cliutil.Parse(...)`
- bind `&cli.Globals` and `appConfig` with `kong.Bind(...)`
- set process-level zerolog formatting:
  - `zerolog.DurationFieldUnit = time.Minute`
  - `zerolog.TimeFieldFormat = time.DateTime + " MST"`
- create the logger from `appConfig.LogLevel`
- call `cliutil.Run(ctx, log)`
- log runtime errors and exit non-zero

The key design point is that `main` stays thin and owns only process-level concerns.

### `cmd/<project>/cli.go`

Defines the root CLI type.

Current structure:

- embeds `cmd.Globals`
- `LogLevel *zerolog.Level`
- `Version kong.VersionFlag`
- `Example example.Command`

Current help text:

- log level: `Log level`
- version: `Output version`
- example subcommand: `Example subcommand for new projects`

Current hook:

- `func (c *CLI) AfterApply(cfg *appconfig.Config) error`

Current `AfterApply` behavior:

1. call `cfg.LoadYAML(c.ConfigFile)`
2. if `--log-level` was provided, override `cfg.LogLevel`

The use of `*zerolog.Level` is intentional so Kong can distinguish:

- flag not provided: `nil`
- flag provided: non-`nil`

## Config Precedence

The generated project currently uses this precedence order:

1. defaults from `appconfig.Default()`
2. YAML loaded from `Globals.ConfigFile`
3. explicit CLI flags

This applies to:

- `log_level` via root CLI `AfterApply`
- any command-specific overrides implemented in future command `AfterApply` hooks

## Rebuild Instructions

If recreating this scaffold from scratch in a new repo, the sequence is:

1. Create `go.mod` with the dependency versions listed above.
2. Create the Makefile with the targets and linker-flag behavior described above.
3. Create `internal/appconfig/config.go` with `Config` and `Default()`.
4. Create `internal/appconfig/load.go` with `LoadYAML(path)`.
5. Create `internal/appconfig/config_test.go` with the config-loading tests.
6. Create `cmd/globals.go` with the `ConfigFile` flag.
7. Create `internal/cli/buildinfo.go`.
8. Create `internal/cli/runner.go`.
9. Create `cmd/<project>/main.go` that:
   - builds `appConfig := appconfig.Default()`
   - binds `&cli.Globals` and `appConfig`
   - creates the logger from `appConfig.LogLevel`
10. Create `cmd/<project>/cli.go` with:
   - embedded `cmd.Globals`
   - `LogLevel *zerolog.Level`
   - `Version kong.VersionFlag`
   - `Example example.Command`
   - `AfterApply(cfg *appconfig.Config) error`
11. Create `cmd/<project>/example/cmd.go`.
12. Create `cmd/<project>/main_test.go`.
13. Create `README.md`.
14. Create `<project>.yaml`.

## Design Rationale

The current design is intentional.

- `internal/cli` exists so future binaries can reuse parser/build-info/logger wiring.
- `internal/appconfig` keeps app config out of `main` and out of the top-level `cmd` package while still making it available to command handlers.
- `cmd/globals.go` holds shared CLI globals that subcommands can import directly.
- `kong.Bind(...)` is used for shared injected objects instead of a custom `RunArgs()` abstraction.
- config loading is explicit Go code rather than Kong’s default config loader, which keeps YAML field naming fully under app control.
- command handlers receive already-merged config and do not need to know how precedence works unless they add command-local overrides.

## Notes For Future Changes

If this scaffold evolves further, the likely update points are:

- `internal/appconfig` if new app-wide config fields are added
- `cmd/<project>/cli.go` if new root flags or root-level overrides are added
- `cmd/<project>/example/cmd.go` if the placeholder command changes
- `cmd/<project>/main_test.go` if parser wiring changes
- `Makefile` if dependency or release workflows change

The most important invariant to preserve is:

- defaults
- then YAML
- then explicit CLI flags

while keeping `main` thin and `internal/cli` focused on runtime wiring rather than app-specific config.
